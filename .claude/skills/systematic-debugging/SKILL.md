# Systematic Debugging

Источники: Superpowers/obra, Microsoft AgentRx (Critical Failure Step), Andrew Ng (Reflection pattern)

## Железный закон

```
НИКАКИХ ФИКСОВ ДО НАХОЖДЕНИЯ КОРНЕВОЙ ПРИЧИНЫ
```

Фикс симптома = провал. «Добавил try/catch и ошибка пропала» = замаскировал проблему, не решил.

## 4 фазы

### Фаза 1: Расследование

1. Прочитай ошибку целиком (стектрейс, логи, вывод)
2. Воспроизведи проблему (если код: запусти, если операционная: восстанови цепочку событий)
3. Найди **Critical Failure Step** (Microsoft AgentRx): ПЕРВЫЙ необратимый шаг, где всё пошло не так. Не последний симптом, а первопричину.
4. Инструментируй границы компонентов (добавь логи между модулями/шагами)
5. Проверь git-историю: что менялось недавно?

Результат фазы: «Проблема возникает в [компонент], когда [условие]. Critical Failure Step: [конкретный шаг].»

### Фаза 2: Анализ паттернов

1. Найди аналогичный РАБОЧИЙ код/процесс
2. Сравни: что отличается между рабочей и сломанной версией?
3. Проверь: это регрессия (работало раньше) или новый дефект?
4. Для операционных проблем: проверь аналогичные успешные случаи

Результат фазы: «Разница между рабочим и сломанным: [конкретно].»

### Фаза 3: Гипотеза

1. Сформулируй гипотезу ПИСЬМЕННО: «Причина в [X], потому что [Y]»
2. Определи минимальное изолированное изменение для проверки
3. Предскажи результат: «Если моя гипотеза верна, то [что именно произойдёт]»

Результат фазы: чёткая гипотеза + предсказание + минимальное изменение для проверки.

### Фаза 4: Фикс

1. Напиши падающий тест (если код), проверь что он падает по правильной причине
2. Сделай ОДИН фикс ОДНОЙ причины (не несколько изменений одновременно)
3. Запусти тест, убедись что проходит
4. Проверь регрессии: все существующие тесты проходят?
5. Для операционных проблем: проверь что аналогичная ситуация теперь обрабатывается правильно

## Stopping conditions

- После **3 неудачных попыток** фикса → СТОП. Поставь под вопрос АРХИТЕКТУРУ, а не продолжай лечить симптомы. Сообщи руководителю.
- Гипотеза не подтвердилась → вернись к Фазе 1, не пробуй следующий фикс наугад.
- Не можешь воспроизвести → собери больше данных, не угадывай.

## Антипаттерны

| Ошибка | Почему плохо | Что делать |
|---|---|---|
| «Быстрый фикс» до расследования | Маскирует причину, проблема вернётся | Пройди все 4 фазы |
| Несколько изменений одновременно | Невозможно понять что помогло | Одно изменение за раз |
| «Добавил try/catch, ошибка пропала» | Замаскировал, не починил | Найди почему ошибка возникает |
| Угадывание вместо анализа | Случайные изменения, потеря времени | Гипотеза → предсказание → проверка |
| Пропуск создания тестов | Нет гарантии что фикс работает | Сначала падающий тест |
| «Работает на моей машине» | Не воспроизведено = не верифицировано | Воспроизведи в чистом окружении |

## Применимость за пределами кода

Этот скилл работает для ЛЮБЫХ проблем, не только технических:

| Проблема | Critical Failure Step | Расследование |
|---|---|---|
| Отчёт ФСИ отклонили | Какой конкретный пункт не прошёл? | Сравнить с успешным отчётом |
| Письмо не получило ответа | Тон? Адресат? Тема? Время отправки? | Сравнить с письмами, на которые ответили |
| Субагент выдаёт плохие результаты | Первый шаг, где результат отклоняется от ожидания | Анализ трассы: intake → контекст → генерация → self-check |
| Процесс регулярно сбоит | Какой шаг процесса нестабилен? | Логировать каждый шаг, найти точку отказа |
